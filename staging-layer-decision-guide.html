Real-World Examples

Example: Facilities Staging (Dynamic Tables)

Characteristics:

  • Simple Single join key (facility_id)
  • Low Conflicts <5% conflict rate
  • Reference Data ~500 facilities
  • Real-Time 30-minute freshness

Implementation:

CREATE OR REPLACE DYNAMIC TABLE dt_stg_facilities
    TARGET_LAG = '30 minutes'
    WAREHOUSE = compute_wh
    REFRESH_MODE = AUTO
AS
WITH oms_latest AS (
    SELECT *
    FROM ods_raw.ods_facilities_source
    WHERE source_system = 'OMS'
      AND ingestion_timestamp >= DATEADD(day, -7, CURRENT_TIMESTAMP())
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY facility_id
        ORDER BY ingestion_timestamp DESC
    ) = 1
),
vems_latest AS (
    SELECT *
    FROM ods_raw.ods_facilities_source
    WHERE source_system = 'VEMS'
      AND ingestion_timestamp >= DATEADD(day, -7, CURRENT_TIMESTAMP())
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY facility_id
        ORDER BY ingestion_timestamp DESC
    ) = 1
)
SELECT
    COALESCE(o.facility_id, v.facility_id) as facility_id,
    -- OMS is primary for facilities
    COALESCE(o.facility_name, v.facility_name) as facility_name,
    COALESCE(o.state, v.state) as state,
    COALESCE(o.phone, v.phone) as phone,
    -- Data quality scoring
    (CASE WHEN facility_id IS NOT NULL THEN 30 ELSE 0 END +
     CASE WHEN facility_name IS NOT NULL THEN 20 ELSE 0 END) as dq_score
FROM oms_latest o
FULL OUTER JOIN vems_latest v
    ON o.facility_id = v.facility_id;

Why Dynamic Tables:

  • ‚úÖ Simple logic (no complex branching)
  • ‚úÖ Automatic refresh (no manual orchestration)
  • ‚úÖ Low conflict rate
  • ‚úÖ Real-time capability

Example: Veterans Staging (Stored Procedures)

Characteristics:

  • Complex Multiple match strategies
  • High Conflicts 30% conflict rate
  • Audit Required Write to reconciliation log
  • Batch Tracking Need batch_id

Implementation:

CREATE OR REPLACE PROCEDURE sp_merge_veterans_to_staging(
    p_batch_id VARCHAR
)
RETURNS VARCHAR
AS
$$
BEGIN
    -- Step 1: Clear existing batch
    DELETE FROM staging.stg_veterans
    WHERE batch_id = :p_batch_id;

    -- Step 2: Merge with complex logic
    INSERT INTO staging.stg_veterans (...)
    SELECT
        xwalk.master_veteran_id,
        -- Complex system-of-record rules
        COALESCE(oms.ssn, vems.ssn) as ssn,
        COALESCE(oms.first_name, vems.first_name) as first_name,
        -- Contact: VEMS preferred (more current)
        COALESCE(vems.email, oms.email) as email,
        -- Disability: OMS authoritative
        COALESCE(oms.disability_rating, vems.disability_rating) as rating,

        -- Conflict detection
        CASE
            WHEN oms.disability_rating != vems.disability_rating
                 AND oms.disability_rating IS NOT NULL
                 AND vems.disability_rating IS NOT NULL
            THEN 'DISABILITY_RATING_MISMATCH'
            ELSE NULL
        END as conflict_type,

        :p_batch_id as batch_id
    FROM reference.ref_entity_crosswalk_veteran xwalk
    LEFT JOIN ods.oms_veterans oms ON ...
    LEFT JOIN ods.vems_veterans vems ON ...;

    -- Step 3: Log conflicts to audit table
    INSERT INTO reference.ref_reconciliation_log (...)
    SELECT
        :p_batch_id,
        'VETERAN',
        master_veteran_id,
        conflict_type,
        oms_value,
        vems_value
    FROM staging.stg_veterans
    WHERE conflict_type IS NOT NULL
      AND batch_id = :p_batch_id;

    RETURN 'Processed: ' || SQLROWCOUNT || ' veterans';
END;
$$;

  • ‚úÖ Complex conflict resolution (30% conflict rate)
  • ‚úÖ Write to audit tables (reconciliation log)
  • ‚úÖ Explicit batch tracking
  • ‚úÖ Multi-step workflow with validation
  • ‚úÖ Error handling with TRY/CATCH

Example: Appointment Events Fact (Dynamic Tables)

Characteristics:

  • Simple Single source (VEMS only)
  • No Conflicts No merging needed
  • Fact Table Transactional events
  • Real-Time 20-minute freshness

Implementation:

CREATE OR REPLACE DYNAMIC TABLE dt_stg_fact_appointment_events
    TARGET_LAG = '20 minutes'
    WAREHOUSE = compute_wh
    REFRESH_MODE = AUTO
AS
WITH vems_appointments AS (
    SELECT *
    FROM ods_raw.ods_appointment_events_source
    WHERE source_system = 'VEMS'
      AND ingestion_timestamp >= DATEADD(day, -7, CURRENT_TIMESTAMP())
    -- Keep only latest version of each event
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY event_id
        ORDER BY ingestion_timestamp DESC
    ) = 1
)
SELECT
    event_id,
    appointment_id,
    -- Enrich with master IDs from crosswalks
    vet_xwalk.master_veteran_id,
    eval_xwalk.master_evaluator_id,
    fac_xwalk.master_facility_id,

    -- Event details
    event_type,  -- SCHEDULED, CONFIRMED, CANCELLED, NO_SHOW
    event_timestamp,
    scheduled_date,

    -- Derive sequence number
    ROW_NUMBER() OVER (
        PARTITION BY appointment_id
        ORDER BY event_timestamp
    ) as event_sequence_number,

    -- Flags
    IFF(event_type = 'NO_SHOW', TRUE, FALSE) as no_show_flag,

    -- Metadata
    'VEMS' as source_system,
    CURRENT_TIMESTAMP() as staging_processed_timestamp
FROM vems_appointments apt
-- Join to crosswalks for master IDs
LEFT JOIN reference.dt_crosswalk_veterans vet_xwalk
    ON apt.veteran_ssn = vet_xwalk.master_veteran_id
LEFT JOIN reference.dt_crosswalk_evaluators eval_xwalk
    ON apt.evaluator_npi = eval_xwalk.master_evaluator_id
LEFT JOIN reference.dt_crosswalk_facilities fac_xwalk
    ON apt.facility_id = fac_xwalk.master_facility_id;

Why Dynamic Tables:

  • ‚úÖ Single source (VEMS) - no merge conflicts
  • ‚úÖ Straightforward extraction + enrichment
  • ‚úÖ Real-time requirement (20 min)
  • ‚úÖ Automatic dependency on crosswalks
  • ‚úÖ No audit table writes needed
